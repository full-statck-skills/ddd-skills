# Boundaries — 微服务边界与架构演进

## 三层边界模型

| 边界类型 | 定义 | 作用 |
|----------|------|------|
| **逻辑边界** | 微服务内聚合之间的边界 | 业务内聚，架构演进的最小单元 |
| **物理边界** | 微服务之间的部署边界 | 运行隔离、独立部署 |
| **代码边界** | 不同聚合/层的代码目录边界 | 代码隔离，方便拆分重组 |

## 逻辑边界

```
微服务 X：
├── 聚合 A ←── 逻辑边界
│   ├── 实体 a1
│   └── 实体 a2
├── 聚合 B ←── 逻辑边界
│   └── 实体 b1
└── 聚合 C ←── 逻辑边界
    └── 实体 c1
```

| 特性 | 说明 |
|------|------|
| 每个聚合内业务高内聚 | 聚合间通过应用层编排，不直接调用 |
| 聚合是微服务拆分的最小单位 | 以聚合目录为单位进行代码迁移 |
| 逻辑边界可升级为物理边界 | 当聚合面临性能瓶颈时独立为微服务 |

## 物理边界

```
微服务 A (独立进程)   微服务 B (独立进程)
    │                     │
    ├── 独立部署          ├── 独立部署
    ├── 独立扩容          ├── 独立扩容
    └── API网关通信 ←──→  └── API网关通信
```

| 何时拆分物理边界 | 何时不拆 |
|-----------------|---------|
| 聚合面临性能瓶颈 | 聚合间有强一致需求 |
| 团队组织边界差异 | 团队微服务管理能力不足 |
| 独立发布频率要求 | 拆分后集成成本 > 收益 |
| 技术异构需求 | 微服务数量已过多 |

## 避免"小单体微服务"

```
错误模式（小单体）：
  微服务 A                微服务 B
  ┌────────────┐         ┌────────────┐
  │ Controller │         │ Controller │
  │  Service   │         │  Service   │
  │   DAO      │         │   DAO      │
  └────────────┘         └────────────┘
  只有物理边界，没有逻辑边界
  代码仍高度耦合，边界模糊

正确模式（DDD 微服务）：
  微服务 A                微服务 B
  ┌────────────┐         ┌────────────┐
  │ interfaces │         │ interfaces │
  │application │         │application │
  │  domain/   │         │  domain/   │
  │  ├─聚合1   │         │  ├─聚合3   │
  │  └─聚合2   │         │  └─聚合4   │
  │infrastructure│       │infrastructure│
  └────────────┘         └────────────┘
  既有物理边界，也有逻辑边界
```

## 以聚合为单位的演进

```
演进前:                             演进后:
微服务 A:                           微服务 A:
├── 聚合 X (热点模块，拖累性能)        ├── 聚合 Y
├── 聚合 Y                           └── 聚合 Z
└── 聚合 Z
                                    微服务 B (新建):
                                    └── 聚合 X (独立部署，独立扩容)
```

### 演进操作步骤

```
1. 复制聚合 X 代码目录到新微服务
2. 调整依赖（聚合 X 的仓储实现等）
3. 独立部署新微服务
4. API 网关添加路由
5. 逐步切流量
6. 原微服务删除聚合 X 代码
```

## 聚合重组的两种模式

### 拆分（一个微服务 → 多个）

```
触发条件：性能瓶颈、独立迭代需求、团队拆分
操作：以聚合为单位迁移代码目录
```

### 合并（多个微服务 → 一个）

```
触发条件：微服务过细、集成成本高、同一团队维护
操作：多个聚合的代码目录合并到一个微服务
```

## 设计原则

| 原则 | 说明 |
|------|------|
| 聚合间零直接依赖 | 通过应用层编排或事件驱动 |
| 代码边界 = 业务边界 | 代码目录结构直接反映领域模型 |
| 演进不破坏逻辑边界 | 拆分/合并以聚合为单位，不影响其他聚合 |
| 先逻辑后物理 | 初期不必拆太多微服务，逻辑边界清晰即可 |

## 常见错误

| 错误 | 纠正 |
|------|------|
| 所有微服务都按功能拆，没有逻辑边界 | DDD 先建模，聚合内高内聚 |
| 微服务拆分过细 | 先建逻辑边界，需要时再拆物理边界 |
| 聚合拆分后忘记删原代码 | 拆分后及时清理，避免双写 |
| 聚合间 Service 直接互相调用 | 必须通过应用层编排 |
