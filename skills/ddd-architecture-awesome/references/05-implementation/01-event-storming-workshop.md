# Event Storming Workshop — 事件风暴工作坊方法论

## 概述

事件风暴 = 团队协作活动，通过头脑风暴罗列领域中所有领域事件 → 聚类分析 → 构建领域模型。

**发明者**: Alberto Brandolini

## 工作坊准备

### 参与角色

| 角色 | 职责 | 必选 |
|------|------|:--:|
| **领域专家** | 提供业务知识，主导事件识别 | ✅ |
| DDD 专家 | 引导工作坊流程，保持聚焦 | 推荐 |
| 产品经理 | 提供用户场景和需求 | ✅ |
| 架构师 | 关注技术可行性和边界 | ✅ |
| 开发人员 | 理解模型，为代码落地做准备 | ✅ |
| 测试人员 | 理解业务流程，设计测试用例 | 推荐 |

### 材料与场地

| 项目 | 说明 |
|------|------|
| 贴纸 | 多色即时贴（橙色=事件、蓝色=命令、绿色=实体、黄色=补充、紫色=热点） |
| 水笔 | 粗头，方便远距离阅读 |
| 场地 | 一面长墙（建议 8 米），无桌椅，可自由走动 |
| 胶带/磁扣 | 便于贴纸随时调整位置 |

## 6 步工作坊流程

### Step 1: 产品愿景 (30 min)

**目标**: 统一团队对产品目标、核心价值、差异化竞争点的认知。

```
产出：产品愿景墙
├── 产品目标用户是谁？
├── 核心价值是什么？
├── 与竞品的差异和优势在哪里？
└── 一段产品愿景描述
```

**参与**: 领域专家、业务需求方、产品经理、项目经理、开发经理

### Step 2: 场景分析 (60 min)

**目标**: 从用户视角探索典型场景，找出领域事件、命令、实体。
**方法**: 用户旅程分析 → 逐环节分析操作和事件。

```
产出：场景分析墙
├── 每个场景的领域事件列表（橙色）
├── 触发事件的命令（蓝色）
├── 补充信息（黄色）：外部依赖、注意事项
└── 角色和参与者
```

### Step 3: 实体提取 (30 min)

**目标**: 从命令和事件中提取产生这些行为的实体。

```
流程：
命令/事件 → 谁产生的？ → 画实体（绿色贴纸）
命令/事件 → 作用于谁？ → 画实体

产出：实体清单 + 实体-事件-命令关系图
```

### Step 4: 聚合识别 (30 min)

**目标**: 根据业务内聚原则，将紧密关联的实体聚类为聚合，确定聚合根。

```
聚合根判断标准：
├── 有独立生命周期？
├── 有全局唯一 ID？
├── 负责创建/修改其他实体？
└── 有专门模块管理？

产出：聚合边界图 + 聚合根清单
```

### Step 5: 限界上下文划分 (30 min)

**目标**: 根据语义边界将聚合归类到限界上下文。

```
原则：
├── 同一上下文中术语含义唯一
├── 业务紧密关联的聚合放同一上下文
└── 不同团队负责的部分考虑分开

产出：限界上下文清单 + 上下文映射关系
```

### Step 6: 微服务拆分 (30 min)

**目标**: 在领域模型基础上，综合考虑非业务因素确定微服务边界。

```
非业务因素：
├── 职责单一性
├── 敏态/稳态业务分离
├── 非功能性需求（性能、安全、弹性）
├── 团队沟通效率
└── 技术异构

产出：微服务边界图 + 微服务清单
```

## Hotspot（热点）管理

工作坊中遇到的不确定问题，用紫色/红色贴纸标记：

```
常见 Hotspot：
├── 术语含义不统一
├── 业务规则不明确
├── 外部系统依赖未知
├── 边界划分有争议
└── 需要进一步调研的问题
```

> 会后必须有专人跟进所有 Hotspot，否则事件风暴成果打折扣。

## 实战案例：用户中台领域建模

### 产品愿景

> 为内外部用户提供统一的登录认证和权限管理能力，支持多渠道接入，实现一次认证多系统通行。

### 场景分析 → 实体提取

| 场景 | 命令 | 事件 | 实体 |
|------|------|------|------|
| 系统设置 | 创建系统、配置菜单 | 系统已创建、菜单已配置 | 系统、菜单 |
| 岗位设置 | 创建岗位、分配权限 | 岗位已创建 | 岗位 |
| 用户创建 | 创建用户、设置岗位 | 用户已创建 | 用户、账户 |
| 登录认证 | 用户登录、验证令牌 | 登录成功、令牌生成 | 认证票据 |

### 聚合识别

| 聚合 | 聚合根 | 包含的实体 |
|------|--------|-----------|
| 系统功能 | 系统 | 菜单 |
| 岗位 | 岗位 | - |
| 用户信息 | 用户 | 用户日志 |
| 账户 | 账户 | - |
| 认证票据 | 认证票据 | - |

### 限界上下文划分

| 上下文 | 包含聚合 | 微服务 |
|--------|----------|:--:|
| 权限域 | 系统功能、岗位 | ✓ |
| 用户信息域 | 用户信息 | ✓ |
| 认证域 | 账户、认证票据 | ✓ |

> **注意**: 用户日志数据量巨大需大数据技术 → 与用户信息技术异构 → 从用户信息服务中拆出独立微服务。

| 产出物 | 格式 | 用途 |
|--------|------|------|
| 产品愿景描述 | 一段文字 | 统一团队方向 |
| 事件时间线 | 贴纸墙照片 | 理解业务流程 |
| 实体-命令-事件关系 | 表格 | 领域建模输入 |
| 聚合边界图 | 图/表格 | 代码分包依据 |
| 限界上下文清单 | 表格 | 微服务边界划分 |
| 上下文映射图 | 图 | 集成关系设计 |
| Hotspot 清单 | 列表 | 会后跟踪解决 |
| 微服务拆分决策 | 表格 | 技术架构输入 |

> 中型项目，完整领域建模约 2 周。
