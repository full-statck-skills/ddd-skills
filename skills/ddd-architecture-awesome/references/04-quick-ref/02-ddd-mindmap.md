# DDD 思维导图 — 知识体系全景

## 一、战略设计

```
DDD 战略设计
├── 统一语言（Ubiquitous Language）
│   ├── 团队共识
│   ├── 代码即文档
│   └── 领域专家参与
├── 限界上下文（Bounded Context）
│   ├── 语义边界
│   ├── 上下文映射
│   │   ├── 合作关系（Partnership）
│   │   ├── 共享内核（Shared Kernel）
│   │   ├── 客户-供应商（Customer-Supplier）

│   │   ├── 开放主机（OHS）
│   │   └── 发布语言（Published Language）
│   └── 演进策略

    ├── 核心域（Core Domain）
    ├── 支撑域（Supporting Subdomain）
    └── 通用域（Generic Subdomain）
```

## 二、战术设计

```
DDD 战术设计
├── 实体（Entity）
│   ├── 唯一标识
│   ├── 可变状态
│   ├── 生命周期连续性
│   └── 业务行为
├── 值对象（Value Object）
│   ├── 无标识
│   ├── 不可变
│   └── 描述性属性
├── 聚合（Aggregate）
│   ├── 一致性边界 = 事务边界
│   ├── 聚合根（Aggregate Root）
│   │   ├── 全局唯一标识
│   │   ├── 访问入口
│   │   └── 协调内部对象
│   └── 设计原则
│       ├── 小聚合
│       ├── ID引用
│       ├── 最终一致性（跨聚合）
│       └── 一次事务一个聚合
├── 领域服务（Domain Service）
│   ├── 无状态
│   ├── 跨实体逻辑
│   └── 不归属任何实体
├── 仓储（Repository）
│   ├── 聚合级别
│   ├── 接口在 Domain
│   └── 实现在 Infrastructure
├── 领域事件（Domain Event）
│   ├── 过去式命名
│   ├── 解耦聚合
│   └── 最终一致性
├── 工厂（Factory）
│   └── 复杂对象创建
└── 规约（Specification）
    └── 可组合的业务规则
```

## 三、架构模式全景

```
DDD 架构模式
├── 分层架构
│   ├── Interface → Application → Domain ← Infrastructure
│   ├── Domain 零依赖
│   └── 严格分层（推荐）：每层只依赖紧邻下层
├── 六边形架构（Ports & Adapters）
│   ├── 端口（Port）：领域定义接口
│   ├── 适配器（Adapter）：技术实现
│   └── 依赖指向核心
├── 整洁架构（Clean Architecture）
│   ├── Entities → Use Cases → Adapters → Frameworks
│   └── 依赖规则：只向内
├── 洋葱架构（Onion Architecture）
│   ├── Domain 在中心
│   ├── 内层定义接口
│   └── 外层实现
├── COLA 架构
│   ├── Adapter → App → Domain ← Infrastructure
│   ├── Command/Query 分离
│   └── 国内工程化标准
└── CQRS
    ├── 命令侧：写操作 + 领域模型
    │   └── 事件溯源（可选）
    └── 查询侧：读操作 + 读模型
        └── 物化视图
```

## 四、设计过程与方法

### 事件风暴（Event Storming）

```
贴纸颜色约定：
├── 橙色 ── 领域事件
├── 蓝色 ── 命令
├── 黄色 ── 聚合
├── 粉色 ── 角色/参与者
├── 紫色 ── 策略/热点
└── 绿色 ── 实体
```

### 领域建模 5 步法

```
1. 发现领域概念（事件风暴 → 实体、命令、事件）
2. 识别实体和值对象（归类领域对象）
3. 定义聚合和聚合根（按一致性边界聚类）
4. 划分限界上下文（按语义边界分组）
5. 建立上下文映射（确定 BC 间关系）
```

## 五、设计原则与模式

### 核心设计原则

| 原则 | 在 DDD 中的体现 |
|------|----------------|
| 单一职责 | 每个聚合只负责一个业务能力 |
| 依赖倒置 | 领域层定义接口，基础层实现 |
| 接口隔离 | Port/Adapter 按需定义，不创建"上帝端口" |
| 开闭原则 | 新增聚合不修改现有聚合 |

### 战术模式

| 模式 | 用途 |
|------|------|
| 仓储模式（Repository） | 持久化抽象，解耦领域层与数据库 |
| 工厂模式（Factory） | 复杂聚合创建 |
| 规约模式（Specification） | 可组合的业务规则 |
| 策略模式（Strategy） | 可替换的业务算法 |

### 集成模式

| 模式 | 适用 |
|------|------|
| RESTful API | 微服务间同步调用 |
| 消息队列 | 跨微服务异步事件 |
| 领域事件 | 聚合间/微服务间解耦 |
| API 网关 | 前端统一入口 |

## 六、实施要点

```
├── 团队协作
│   ├── 领域专家深度参与（首要成功因素）
│   ├── 统一语言建立（代码 = 文档 = 沟通）
│   └── 持续沟通（事件风暴工作坊）
├── 技术实现
│   ├── 领域层纯净（零框架依赖）
│   ├── 基础设施隔离（依赖倒置）
│   └── 测试策略
│       ├── 单元测试 → 领域对象
│       ├── 集成测试 → 仓储/适配器
│       └── 验收测试 → 完整用例
└── 演进策略
    ├── 增量式实施（先核心域，再扩展）
    ├── 以聚合为单位演进（拆分/组合）
    └── 模型持续精化（领域服务下沉合并）
```

## 七、适用场景与决策

```
是否使用 DDD？
├── 适用 ✓
│   ├── 复杂业务领域（规则、状态机、约束）
│   ├── 长生命周期系统（> 2 年）
│   ├── 团队规模 5+ 人
│   └── 需要业务创新和快速响应
├── 谨慎 ~
│   ├── 中等复杂度
│   ├── 无专职领域专家
│   └── 团队 DDD 经验不足（从分层架构开始）
└── 不适用 ✗
    ├── 简单 CRUD 应用
    ├── 技术验证/原型项目
    └── 一次性短周期项目
```

## 八、关键成功因素

1. **领域专家深度参与** — 没有领域专家的 DDD = 为虚无建模
2. **团队对 DDD 的理解和认同** — 统一方法论，减少内耗
3. **持续重构的勇气** — 领域模型随业务演进，不惧怕调整
4. **合适的项目复杂度** — DDD 不是银弹，简单项目用简单方法
5. **良好的技术实现支撑** — DevOps、CI/CD、监控体系
