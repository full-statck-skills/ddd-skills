# Distributed Architecture FAQ — 分布式架构关键设计 10 问

## Q1: 如何选择分布式数据库？

| 方案 | 代表产品 | 适用 |
|------|----------|------|
| **一体化分布式数据库** | OceanBase, GaussDB | 大厂，云计算底座，超强处理能力 |
| **集中式DB + 中间件** | MyCat+MySQL, TBase | 中大型企业，成本适中 |
| **集中式DB + 分库类库** | ShardingSphere | 简单读写场景，成本最低 |

## Q2: 如何设计分库主键？

> **首选客户 ID 作为分库主键**，确保同一客户数据在同一数据单元。

| 原则 | 说明 |
|------|------|
| 避免跨数据单元访问 | 跨库查询严重影响性能 |
| 支持客户一致性服务 | 同一客户数据在同一单元 |
| 可选替代 | 机构 ID、用户 ID 等 |

## Q3: 数据库同步与复制？

| 方式 | 时效性 | 适用 |
|------|:--:|------|
| ETL/定时提数 | 小时级 | 离线报表 |
| CDC (Debezium等) | 准实时 | 业务数据同步、事件驱动 |

> CDC = 数据库逻辑日志增量捕获，实现数据复制与应用解耦。

## Q4: 跨库关联查询？

| 场景 | 方案 |
|------|------|
| 跨微服务主题查询（如客户全景视图） | 建设**主题数据库**，CDC 汇集数据为宽表 |
| 表与表关联（如机构表+业务表） | **小表广播**：在业务库冗余代码副表，事件驱动同步 |

## Q5: 高频热点数据？

| 方案 | 适用 |
|------|------|
| **Redis 缓存** | 代码表、商品信息等高频读取 |
| **ElasticSearch** | 需要模糊搜索的高频数据 |

## Q6: 前后序业务数据？

```
前序微服务 → 当前微服务 → 后序微服务
  投保          保单          理赔
```

| 数据关联方式 | 说明 |
|-------------|------|
| 通过领域事件将前序数据冗余到当前微服务 | 事件携带前序数据，当前服务持久化 |
| 设计为值对象 | 只读、整体替换、不需要查询 |
| 设计为实体 | 多条数据、需要查询和统计分析 |

## Q7: 数据中台建设？

```
Step 1: 数据汇集 — 统一标准，汇集各微服务和渠道数据
Step 2: 数据加工 — 按主题/场景加工，建立客户视图、渠道视图等
Step 3: 数据赋能 — 萃取价值，支持业务创新
```

> 数据中台不仅限分析场景，也适用于交易场景（基于 CDC 的准实时数据）。

## Q8: BFF 与企业级编排？

```
BFF 微服务 (无领域模型)
├── 跨中台服务编排
├── 前端渠道适配
└── 与前端版本协同发布

vs

应用服务 (微服务内)
└── 微服务内聚合服务的编排
```

| 对比 | BFF | 应用服务 |
|------|-----|----------|
| 位置 | 中台之上 | 微服务内 |
| 编排范围 | 跨中台微服务 | 微服务内聚合 |
| 是否有领域模型 | 无 | 调用领域层的服务 |

## Q9: 分布式事务 vs 事件驱动？

| 方式 | 一致性 | 性能代价 | 适用 |
|------|:--:|:--:|------|
| **分布式事务** (XA/TCC/Saga) | 强一致 | 高（性能代价大） | 实时性要求极高、不可避免 |
| **领域事件 + 最终一致** | 最终一致 | 低 | 绝大多数跨微服务场景 |

> 优先用领域事件。分布式事务是最后手段。

### 事件驱动的额外收益

- 解耦微服务
- 削峰填谷，减轻数据库压力
- 读写分离
- 一对多发布（一个发布方 N 个订阅方）

## Q10: 多中心多活？

| 设计要点 | 说明 |
|----------|------|
| **分布式数据库** | 支持多数据中心部署、数据多副本 |
| **单元化架构** | 若干应用组成业务单元作为部署基本单位，单元自包含 |
| **访问路由** | 接入层、应用层、数据层路由，确保流量准确到达 |
| **全局配置管理** | 各数据中心配置实时同步 |

单元化原则：
- 所有业务流程可在本单元完成
- 任意单元数据在多中心有副本
- 单元故障不影响其他同类单元
- 尽量避免跨数据中心调用
