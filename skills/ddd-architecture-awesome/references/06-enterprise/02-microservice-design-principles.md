# Microservice Design Principles — 微服务设计原则与策略

## 演进策略

| 策略 | 做法 | 适用 |
|------|------|------|
| **绞杀者 (Strangler)** | 逐步剥离业务能力，用新微服务替代旧单体 | 大型单体系统演进 |
| **修缮者** | 维持整体不变，局部优化（拆分性能瓶颈模块等） | 系统整体健康，局部有问题 |
| **另起炉灶** | 推倒重做，新系统并行建设后切换 | 系统严重老化，维护成本 > 重建（不推荐核心系统） |

## 四大设计原则

### 原则 1：领域驱动设计，而非数据驱动或界面驱动

```
错误流程: 设计表 → 设计实体 → 实现功能
正确流程: 领域建模 → 确定边界 → 设计微服务 → 映射数据

核心逻辑: 先建领域模型，确定逻辑和物理边界后，再拆微服务。
          外部需求从外到内逐级消化，降低对核心领域层的影响。
```

### 原则 2：边界清晰的微服务，而非泥球小单体

```
泥球小单体:
  只有物理边界，没有逻辑边界
  代码仍高度耦合，需要变更时牵一发动全身

DDD 微服务:
  有逻辑边界（聚合）+ 物理边界（微服务）
  聚合间代码松耦合，通过应用层编排或事件驱动
  变更影响可控，架构演进成本低
```

### 原则 3：职能清晰的分层

```
各层只能调用紧邻下层：
interfaces → application → domain → (domain 接口)
                                    ↓
                              infrastructure (实现)
不能跨层调用：interfaces 不能直接调 domain service
```

| 规则 | 说明 |
|------|------|
| 应用层不写核心业务逻辑 | 只做编排 |
| 领域层不依赖框架 | 零 Spring/JPA import |
| 可复用能力向下层沉淀 | 应用层编排→发现可下沉→领域服务 |

### 原则 4：做自己能 hold 住的微服务

| 过度拆分的代价 | 先建逻辑边界的收益 |
|---------------|-------------------|
| 集成成本上升 | 初期单体也行 |
| 运维成本上升 | 逻辑边界清晰，随时可拆 |
| 监控和定位问题成本上升 | 拆分不费时费力 |
| 团队管理能力跟不上 | 等技术能力提升后再拆 |

## 微服务拆分考量因素

| 因素 | 说明 |
|------|------|
| **业务边界（领域模型）** | 首要依据，按职责单一性和功能完整性拆分 |
| **需求变更频率** | 频繁变动的敏态业务与稳态业务分离 |
| **应用性能** | 性能要求高的功能独立，避免拖累整体 |
| **组织架构** | 康威定律：微服务边界对齐团队边界，团队 10-12 人为宜 |
| **安全边界** | 有特殊安全要求的功能独立拆分 |
| **技术异构** | 不同技术栈的功能按技术边界拆分 |

## DDD 使用误区

| 误区 | 纠正 |
|------|------|
| **所有领域都用 DDD** | 先聚焦核心域，简单 CRUD 用传统方法 |
| **全部采用 DDD 战术设计** | 查询统计等贫领域模型用传统 SQL 方法效率更高 |
| **重战术轻战略** | 战略设计（领域建模）决定微服务边界质量，比战术设计更重要 |
| **DDD 只适用于微服务** | DDD 同样适用于单体应用 |

## 微服务 vs 单体决策

```
问自己：
1. 业务逻辑是否复杂（规则、状态机）？
   ├── NO → 单体可能更好
   └── YES → 2

2. 团队是否有微服务管理能力（DevOps、监控、CI/CD）？
   ├── NO → 先建逻辑边界，逐步演进
   └── YES → 可以拆分微服务

3. 拆分后的集成成本是否低于收益？
   ├── NO → 暂不拆分
   └── YES → 拆分
```
